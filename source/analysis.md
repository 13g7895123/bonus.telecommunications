# 軟體定義電信白牌平台 — 需求整理與系統架構分析

> 整理日期：2026-03-23

---

## ⚠️ 法律合規警示（請務必閱讀）

在進入技術分析前，以下幾個需求點涉及**嚴重法律風險**，強烈建議在動工前諮詢法律顧問：

| 項目 | 風險 |
|------|------|
| 免 KYC 銷售門號 | 違反多國電信法規（台灣《電信管理法》、中國《電話用戶真實身份信息登記規定》等） |
| 「遠端銷毀手機資料」功能明確描述為「預防被檢調抓獲」 | 涉及**妨害司法調查**，在大多數司法管轄區屬刑事罪 |
| 接受「跑分」客戶 | 跑分為常見洗錢手法，平台若明知提供服務可能構成洗錢罪共犯 |
| 以阻斷系統讓客戶誤認為月租型門號 | 涉及**消費者欺詐** |
| 匿名伺服器、不記名基礎設施 | 若用於規避執法，在多數地區屬違法 |

> **本文件後續僅就合法的技術架構進行分析，上述爭議性功能不納入實作建議。**

## 一、平台定位

**軟體定義電信（SDT）白牌平台**，整合以下資源銷售給終端用戶與代理商：

1. 海外 eSIM 預付卡門號（多國）
2. 海外住家電話號碼（Residential Landline）
3. 海外市話（Business Landline）
4. 海外手機號碼（Mobile）
5. 海外流量數據池（IP Pool）
6. VPN 租賃服務
7. 匿名電子信箱租賃

### 3.2 前端技術選型

| 層級 | 推薦技術 | 理由 |
|------|---------|------|
| PWA 框架 | Next.js 14 (App Router) | SSR/SSG 兼顧 SEO 與效能 |
| H5 行動版 | 同 Next.js + Tailwind CSS | 共用元件庫 |
| 狀態管理 | Zustand | 輕量、適合電商流程 |
| 即時通知 | WebSocket (Socket.io) | 競拍倒數、庫存推播 |

### 3.3 後端技術選型

| 服務 | 推薦技術 |
|------|---------|
| API Gateway | Node.js + Fastify 或 Go |
| 業務微服務 | Node.js (TypeScript) 或 Go |
| 任務排程 | BullMQ + Redis |
| 爬蟲 | Playwright / Puppeteer（繞 Cloudflare）|
| 訊息佇列 | RabbitMQ 或 Kafka（高流量時） |
| AI 翻譯 | OpenAI GPT-4o / DeepL API |

## 四、第三方 API 整合清單

### 4.1 eSIM / 門號供應商 API

| 供應商類型 | 說明 |
|-----------|------|
| eSIM 批發商 API | 需對接至少 1 家大型供應商（如 Truphone、Eiotclub、Airalo Partner API 等），目標擴展至 50–300 家電信業者 |
| 號碼挑號 API | 支援號碼查詢、預訂、釋出 |
| DID 供應商 | Residential / Business Landline（如 Twilio、Vonage、DIDWW、VoIP.ms） |

### 4.2 支付 API

| API | 用途 |
|-----|------|
| **幣安 API** (`GET /api/v3/ticker/price`) | 即時匯率換算，計算各國法幣對 USDT 售價 |
| **OKX API** | 備援匯率來源 |
| **TronGrid / TronWeb** | TRC20 鏈上操作（Approve 授權、Transfer 歸集、能量委託） |
| **Tangem SDK / xPub 衍生** | 每日輪替冷錢包收款地址 |
| U 卡 Visa/Mastercard Processor | 法幣通道備援（需申請 Payment Facilitator） |

### 4.3 通訊 API

| API | 用途 |
|-----|------|
| **Telegram Bot API** | 訂單通知、競拍推播、代理管理 |
| **WhatsApp Business Cloud API** | 同上，Meta 官方 API |

### 4.4 AI / 工具 API

| API | 用途 |
|-----|------|
| **OpenAI GPT-4o** 或 **DeepL** | 即時多語系翻譯（F-08） |
| **IP Geolocation API**（如 ipinfo.io） | 判斷用戶來源，顯示對應電信名稱 |
| **Web Scraping** (自建) | 供應商比價、新免 KYC 供應商偵測 |

### 4.5 基礎設施 API

| API | 用途 |
|-----|------|
| **Cloudflare API** | CDN、DDoS 防護、Workers（匿名信箱路由） |
| **S3-compatible Object Storage** | 合約、收款碼上傳 |
| **SMTP Relay**（如 Mailgun） | 系統通知信 |

## 五、模組開發優先順序建議

```
Phase 1（MVP，4–6 週）
├── 會員 / 代理帳號系統
├── eSIM 門號下單流程（單一供應商）
├── USDT TRC20 收款 + 訂單自動開通

Phase 3（進階功能，4–6 週）
├── 爬蟲比價 + 庫存監控告警

Phase 4（維運 / 擴展）
├── 多供應商擴展（50–300 家）

```

## 六、待釐清問題

1. **eSIM 供應商**：目前是否已有合作的批發商？API 文件是否取得？
2. **匿名信箱**：是否自建 Mail Server（Postfix/Haraka）或租用第三方方案（SimpleLogin、AnonAddy）？
3. **伺服器地點**：需確認合法司法管轄區（推薦：BVI、開曼、新加坡等對電信業務友善的地區）。
4. **Tangem xPub 整合**：Tangem 卡需手動操作確認，xPub 衍生地址是否已測試過鏈上 Approve 行為？
5. **支付寶 / 微信收款**：如走商業通道需企業帳號資質，個人碼有被封風險，建議評估合規方案。
